# CodePilot1C EDT — проектная конфигурация QA и совместимость с моделями

Сборка: `0.1.7.20260316-0455`

Два крупных изменения в релизе: QA-настройки теперь живут в проекте, а не в локальных preferences EDT, а provider-слой стал заметно стабильнее на разных моделях и OpenAI-compatible endpoint'ах.

---

## QA / Vanessa Automation привязана к проекту

Раньше настройки Vanessa Automation жили только в локальных preferences EDT. Это создавало типичные проблемы: на другой машине всё приходилось настраивать заново, в Docker или через MCP настройки не подхватывались, а проект нельзя было просто передать коллеге вместе с рабочей конфигурацией.

Теперь QA-настройки хранятся в `tests/qa/qa-config.json` рядом с проектом и могут попадать в Git.

Что можно хранить в проекте:

- путь к `va.epf`
- путь к базовому конфигу Vanessa Automation
- каталоги `features`, `steps`, `results`
- флаги скриншотов при падении
- JUnit-отчёты
- закрытие тест-клиента после прогона
- тихую установку расширений и другие базовые параметры запуска

В свойствах проекта появилась страница `QA / Vanessa Automation`, где эти параметры можно редактировать через UI. При этом проектные настройки имеют приоритет над локальными preferences, что особенно важно для headless, Docker и MCP-сценариев.

### Миграция старых конфигов

Если в проекте уже есть `qa-config.json` старого формата, плагин больше не перезаписывает его молча. Вместо этого он предупреждает о legacy-конфиге и предлагает явную миграцию с сохранением backup.

### Умнее диагностика таймаутов

Если `qa_run` упирается в timeout, теперь возвращается не просто ошибка, а дополнительный контекст:

- какие процессы ещё живы
- продолжают ли расти логи
- заняты ли порты test client
- что осталось после попытки остановки

Это помогает разбирать зависшие прогоны и агенту, и пользователю, не сводя всё к слепому retry.

## Совместимость с OpenAI-compatible моделями

Разные модели по-разному ведут себя при `stream + tools`. Одни сначала отправляют reasoning-блоки, другие фрагментируют `tool_calls`, третьи плохо переносят streaming tool requests вообще. Раньше это приводило к ложным таймаутам, пустым ответам и нестабильным вызовам инструментов.

Provider-слой переработан: теперь transport, нормализация stream chunks, сборка tool calls и policy совместимости разведены по отдельным слоям.

### Политика совместимости моделей

Для разных моделей теперь используется разная стратегия:

- **GLM-5** — tool requests по умолчанию выполняются в `non-stream`
- **MiniMax-M2.5** — tool requests по умолчанию выполняются в `non-stream`
- **Kimi K2.5** — при tool use отключается thinking mode, а большие follow-up запросы после tool result автоматически переводятся в `non-stream`
- **Qwen** — продолжает работать со streaming tool calls

### Что улучшилось технически

- reasoning-first streaming больше не считается ошибкой
- фрагментированные `tool_calls` корректно собираются по частям
- большие tool results больше не отправляются в модель целиком без ограничений
- для крупных follow-up запросов увеличивается timeout и применяется более безопасный execution mode

Это особенно важно для mixed-model endpoint'ов вроде DashScope Coding Plan, где разные модели ведут себя по-разному даже в рамках одного OpenAI-compatible API.

## Итого

- QA-конфиг теперь живёт в проекте и воспроизводим на любой машине
- старые QA-конфиги больше не перетираются молча
- provider-слой стал устойчивее на GLM-5, MiniMax, Kimi и Qwen
- большие tool results и длинные follow-up запросы больше не приводят к ложным streaming timeout там, где это можно обойти policy-уровнем

CodePilot1C EDT становится менее зависимым от локального окружения и предсказуемее как для QA-автоматизации, так и для работы с разными LLM-провайдерами.
