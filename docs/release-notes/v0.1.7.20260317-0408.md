# CodePilot1C EDT — прозрачный QA-конфиг и мягкая проверка шагов

Сборка: `0.1.7.20260317-0408`

Багфикс-релиз для QA-потока. Он закрывает две практические проблемы: было непонятно, откуда плагин берёт пути к Vanessa Automation и её базовому конфигу, а предварительная проверка шагов могла ложно блокировать запуск тестов.

---

## Понятный QA-конфиг

Раньше при запуске тестов было неочевидно, откуда взят путь к `VanessaAutomation.epf`: из project config, из preferences или из fallback. Особенно это мешало в Docker и MCP-сценариях, где локальных preferences может не быть вовсе.

Теперь:

- `qa-config.json` явно хранит ключевые пути: `epf`, базовый конфиг Vanessa Automation, каталог шагов, каталоги `features`, `steps`, `results`
- новый инструмент `qa_init_config` генерирует начальный конфиг проекта
- новый инструмент `qa_explain_config` показывает effective-конфигурацию и источник каждого значения
- `qa_status` возвращает effective-пути вместе с источниками

То есть и агент, и разработчик теперь видят, что именно используется и почему.

---

## Неизвестные шаги больше не блокируют запуск

Встроенный каталог шагов — это только снапшот. Реальная Vanessa Automation может знать больше шагов, чем лежит в JSON-каталоге плагина. Раньше при таком расхождении плагин мог остановить запуск, хотя сами тесты в Vanessa отработали бы нормально.

Теперь есть три режима проверки:

- **off** — не проверять
- **warn** — предупредить, но не блокировать
- **strict** — блокировать как раньше

По умолчанию используется `warn`.

Окончательным источником истины теперь считается сама Vanessa Automation, а не предварительный каталог шагов. Каталог по-прежнему помогает агенту подбирать шаги и анализировать feature-файлы, но больше не мешает запуску тестов по умолчанию.

---

## Единый контракт QA tools

Логика работы с каталогом шагов, базовым конфигом Vanessa и режимом проверки выровнена между всеми QA tools:

- `qa_status`
- `qa_run`
- `qa_validate_feature`
- `qa_steps_search`
- `qa_compile_feature`

Это убирает ситуацию, когда один инструмент видит одну конфигурацию, а другой использует другой fallback и даёт другой результат.

---

## Итого

- QA-конфиг стал прозрачным: видно, что используется и откуда
- ложные блокировки по `unknown_steps` устранены
- Docker и MCP-сценарии стали предсказуемее и меньше зависят от локальных preferences
