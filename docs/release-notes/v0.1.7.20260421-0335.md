# CodePilot1C EDT — hotfix: connect_infobase, GitMutateTool schema, диагностика ошибок

Сборка: `0.1.7.20260421-0335`

Хотфикс к релизу `0.1.7.20260420-0315`. Закрыты три бага из issue #31: ошибка схемы `git_mutate` на Anthropic API, пустой `message` в ответах `connect_infobase`, и NPE при сохранении настроек доступа к файловой инфобазе.

---

## GitMutateTool: схема совместима с Anthropic API

Инструмент `git_mutate` падал на провайдере Anthropic с ошибкой валидации: top-level `allOf` с вложенными `if/then/anyOf` в JSON-схеме не принимается `input_schema` у Claude.

Что сделано:

- Убран top-level `allOf` с условной валидацией аргументов
- Правила "когда какой параметр обязателен" перенесены в runtime — `validateArguments(GitOperation, Path, Map<String,Object>)`
- В схеме остались только `required`, `enum`, `additionalProperties` — формат, который принимают все провайдеры
- Новый тест `publishedSchemaHasNoTopLevelCompositionKeywords` фиксирует отсутствие `allOf`/`anyOf`/`oneOf` на верхнем уровне — регрессия не пройдёт

Инструмент снова вызывается у моделей Anthropic, поведение для OpenAI/Qwen/Kimi не изменилось.

## ConnectInfobaseTool: видимые ошибки вместо пустой строки

Раньше при падении `connect_infobase` ответ выглядел так:

```json
{ "success": false, "error_code": "EDT_SERVICE_UNAVAILABLE", "message": "" }
```

Причина — исключение с `getMessage() == null` терялось в catch-блоке. Теперь:

- Полный стектрейс пишется в лог через `LOG.error(..., e)` с `op_id` и именем проекта
- Если `e.getMessage()` пустой или `null`, в `message` подставляется `ClassName at TopStackFrame` (например `NullPointerException at EdtInfobaseConnectService.storeAccessSettings:391`)
- Новый тест `nullMessageExceptionSurfacesClassNameInstead` гарантирует, что `message` никогда не пустой

Разработчики видят реальную причину без необходимости воспроизводить в дебаггере.

## persistReference: гарантированный UUID перед storeSettings

Файл-инфобаза падала на шаге `storeAccessSettings` с NPE, которая проявлялась как `"Failed to store infobase access settings: null"`.

Корневая причина — `persistReference()` имел два ранних return'а (когда `!isPersistenceSupported()` или инфобаза уже зарегистрирована), после которых `InfobaseReference.getUuid() == null`, а EDT'шный `IInfobaseAccessManager.storeSettings` внутри себя NPE'ил по null-UUID.

Что исправлено:

- `persistReference()` теперь гарантирует UUID на возвращаемой ссылке на всех ветках:
  - persistence не поддерживается → локально присваивается `UUID.randomUUID()`
  - инфобаза уже зарегистрирована → UUID копируется с существующей записи (через `findInfobaseByName`/`findInfobaseByUuid`), fallback на случайный
  - обычная ветка `manager.add()` → defensive fallback, если EDT не проставил
- `storeAccessSettings()` получил pre-flight проверку: если UUID всё же `null`, бросается понятная ошибка `EDT_SERVICE_UNAVAILABLE` с явным текстом вместо NPE из недр EDT
- Catch-блок использует тот же class-name fallback, если внутренняя ошибка пришла с пустым сообщением

Добавлены тесты `persistReferenceAssignsUuidWhenPersistenceUnsupported` и `storeAccessSettingsSurfacesClassNameWhenMessageNull`.
