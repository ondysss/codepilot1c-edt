# CodePilot1C EDT — запуск приложения из EDT, агентный QA workflow и анализ ошибок инструментов

Сборка: `0.1.7.20260309-1749`

> Актуализация на 2026-03-25: текущий public tool surface использует composite tools `edt_diagnostics(command=...)`, `qa_inspect(command=...)` и `qa_generate(command=...)`. Ниже названия из исходного релиза приведены к текущим публичным эквивалентам.

В этой версии CodePilot1C получил полноценный EDT runtime workflow для агента: теперь он может обновить связанную инфобазу, запустить приложение проекта через `RuntimeClient.launch`, проверить QA-окружение, подготовить контекст формы, собрать и валидировать feature-сценарий, а также разобрать ошибки инструментов в структурированном виде. Параллельно добавлен локальный E2E/MCP runbook для smoke-проверок через поднятый EDT MCP Host.

---

## EDT runtime для агента

### Новые инструменты

Добавлены два новых built-in tool для работы через EDT runtime:

1. **`edt_diagnostics(command=update_infobase)`**  
   Обновляет инфобазу, связанную с EDT-проектом. Поддерживает `dry_run`, возвращает структурированный JSON с `op_id`, `status`, `project_name`, `updated` и данными о связанной инфобазе.

2. **`edt_diagnostics(command=launch_app)`**  
   Запускает приложение проекта по `RuntimeClient.launch`. Поддерживает `dry_run`, `wait_for_exit`, `timeout_s`, `additional_parameters`, возвращает `pid`, `runtime_version`, путь к launch config, log path и команду запуска.

### Что сделано под капотом

- Вынесен общий EDT runtime backend: резолв проекта, связанной инфобазы, launch configuration и thick client.
- Добавлен registry фоновых launch-процессов с очисткой при остановке плагина.
- `qa_run` переведен на общий merge access settings, чтобы не дублировать EDT launch-логику.
- Все EDT runtime ошибки нормализованы в стабильные `error_code` (`PROJECT_NOT_FOUND`, `LAUNCH_CONFIG_NOT_FOUND`, `RUNTIME_NOT_RESOLVED`, `PROCESS_TIMEOUT` и др.).

### Зачем это нужно

Раньше агент мог работать с EDT-метаданными и диагностикой, но не умел завершать полный цикл "обновить базу -> запустить приложение". Теперь это можно делать прямо из чата и через внешний MCP-клиент.

## QA workflow для агентного E2E

### Новый поток работы

В плагине появился полный pipeline для подготовки и запуска сценариев Vanessa Automation:

- **`qa_inspect(command=status)`** — проверка окружения и EDT runtime readiness
- **`qa_prepare_form_context`** — подготовка формы для QA, включая auto-create default form при отсутствии
- **`qa_plan_scenario`** — построение структурированного сценарного плана из цели
- **`qa_generate(command=compile_feature)`** — компиляция structured plan в канонический `.feature`
- **`qa_validate_feature`** — валидация feature до запуска
- **`qa_run`** — запуск Vanessa Automation с интеграцией в EDT runtime
- **`qa_inspect(command=steps_search)`** — debug-поиск по каталогу шагов

### Дополнительно

- Добавлен step registry и связанный QA model layer.
- Подготовлены unit tests для QA config, feature compiler, step registry и form-context flow.
- В prompts и agent profiles зафиксирован обязательный workflow: сначала `qa_inspect(command=status)`, затем подготовка контекста, затем компиляция/валидация, потом запуск.

## Анализ ошибок инструментов

### Новый инструмент

Добавлен **`edt_diagnostics(command=analyze_error)`** — diagnostic command, который разбирает ошибки других инструментов и возвращает:

- `error_code`
- краткое summary
- вероятные причины
- рекомендуемые действия
- suggested tool calls
- `log_tail`, если ошибка ссылается на лог-файл

### Что поддерживает

Инструмент понимает:

- JSON-ошибки с `error_code`
- JSON-ошибки с `error`
- legacy формат `[ERROR_CODE] ...`
- строки вида `ERROR_CODE: ...`

Плюс `ToolGraphRouter` теперь тоже умеет извлекать `error_code` из новых JSON payload и корректнее маршрутизирует дальнейшие шаги агента после неуспешных tool call.

## Локальный MCP / E2E runbook

Для локальной проверки через реальный EDT MCP Host добавлены:

- `tools/run-edt-e2e-local.sh`
- `tools/run-qwen-local-edt-suite.sh`
- `tools/run-qwen-mcp-suite.py`
- набор сценариев и suite-файлов в `evals/qwen/`

Это позволяет локально прогнать цепочку:

`build -> p2 update -> relaunch EDT -> MCP smoke -> qa_inspect(command=status) -> qa_run`

и проверить, что новые инструменты доступны и работают через входящий MCP Host, а не только в unit tests.

## Что важно для пользователей

- Агент теперь умеет не только менять метаданные, но и запускать проект через EDT runtime.
- QA-цикл стал воспроизводимым и формализованным: меньше ручных шагов, меньше “галлюцинаций” в `.feature`.
- Ошибки инструментов стали пригодны для повторного машинного анализа и автоматического восстановления сценария.
- Все новые возможности доступны и внутри EDT, и через внешний MCP-клиент при подключении к локальному MCP Host.
